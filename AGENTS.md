# AGENTS.md

## 1. 项目协作原则
- 每次只完成一个小任务。
- 不允许一次性实现多个业务模块。
- 不允许跨业务范围修改无关代码。
- 如果任务需要修改超出指定范围的文件，必须先在回复中说明原因，不得直接修改。
- 不允许无关重构。
- 不允许随意新增依赖。
- 不允许删除现有可运行逻辑。
- 不允许为了“优化”而改变已有目录职责。
- 不允许在没有明确要求时修改路由、全局样式、状态管理或构建配置。

## 2. 修改前阅读规则
每次修改前必须优先阅读：
- README.md
- AGENTS.md
- docs/frontend-architecture.md
- docs/roadmap.md

如果任务涉及已有代码，必须先阅读相关模块文件，再进行修改。

## 3. 代码修改边界
- 修改页面时，只修改对应 view 文件及其必要的局部组件。
- 修改布局时，只修改 `src/layout` 相关文件。
- 修改菜单时，只修改 `src/config/menu.ts`。
- 修改路由时，只修改 `src/router` 相关文件。
- 修改状态管理时，只修改 `src/stores` 下对应 store。
- 修改类型时，只修改 `src/types` 下对应文件。
- 修改接口请求时，只修改 `src/api` 或 `src/services` 下对应文件。
- 修改全局样式时，只修改 `src/styles` 下文件，并说明原因。
- 不允许一个任务同时修改多个无关业务模块。

## 4. Vue 组件规范
- 页面组件统一放在 `src/views`。
- 布局组件统一放在 `src/layout`。
- 通用组件后续统一放在 `src/components`。
- 组件命名使用 PascalCase。
- 页面文件命名建议使用 `XxxView.vue`。
- 不要在页面中堆积过多业务逻辑。
- 后续复杂模块应拆分为局部组件。

## 5. TypeScript 规范
- 不要使用 `any`，除非有明确说明并解释原因。
- 业务数据类型应放在 `src/types`。
- API 返回结构应定义类型。
- mock 数据也应尽量使用类型约束。
- 不要通过关闭类型检查来规避报错。

## 6. 样式规范
- Element Plus 负责复杂组件，如表格、表单、弹窗、分页、菜单。
- 全局 CSS 只放基础布局、reset、通用页面类和主题变量。
- 页面业务样式优先使用 scoped CSS。
- 不要把业务页面样式大量写入全局 `index.css`。
- 后续如接入 Tailwind，仅用于布局、间距、圆角、阴影等微调。
- Tailwind 不替代 Element Plus。
- 不要为了样式调整破坏整体布局。

## 7. 文档同步规则
任何代码修改完成后，都必须检查文档是否需要更新。

每一次代码修改，都必须在 `docs/development-log.md` 追加开发记录。

当以下内容变化时，必须更新对应文档：

- 新增、删除、重命名目录或文件：
  - 更新 `docs/frontend-architecture.md`
  - 必要时更新 README.md 的目录结构说明
- 新增或调整页面、路由、菜单：
  - 更新 README.md 的当前功能或计划功能
  - 更新 `docs/frontend-architecture.md` 中相关模块说明
  - 更新 `docs/development-log.md`
- 新增业务功能：
  - 更新 README.md 的当前已完成内容
  - 更新 `docs/roadmap.md` 的进度状态
  - 更新 `docs/development-log.md`
- 新增依赖、修改启动方式、修改构建方式：
  - 更新 README.md 的技术栈与启动方式
  - 更新 `docs/development-log.md`
- 只是修复 bug 或样式问题：
  - 至少更新 `docs/development-log.md`

## 8. 文档真实性规则
- 不允许把计划中功能写成已完成。
- 不允许编造不存在的接口、页面或能力。
- 如果功能使用 mock 数据，必须明确标注“当前为 mock 数据”。
- 如果后端尚未接入，必须明确标注“后端待接入”。
- 如果只是页面骨架，必须明确标注“页面骨架已完成，业务逻辑待实现”。

## 9. 验证规则
- 修改完成后，应优先执行 `npm run build` 验证 TypeScript 与构建是否通过。
- 如果无法执行命令，必须说明原因。
- 涉及页面样式或交互时，需要说明人工验证路径，例如访问 `/dashboard`、`/prompts`。

## 10. 每次任务回复格式
以后每次完成任务后，必须按以下格式回复：
1. 代码改动文件
2. 文档改动文件
3. 本次功能状态
4. 是否存在 mock / 待实现内容
5. 验证方式
6. 下一步建议
7. 建议commit的标题（中文）
