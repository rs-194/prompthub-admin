# 模块设计流程规范

## 1. 文档目的

本文档用于约束新业务模块的设计流程。

- 任何复杂模块在写代码前，应该先明确数据流、状态设计、函数职责和组件通信方式。
- Codex 不应直接凭功能描述自由发挥实现模块。
- 模块实现应先形成可审核的方案，再进入代码阶段。

## 2. 标准开发流程

1. 明确模块目标
2. 输出模块实现方案，不写代码
3. 人类审核并确认方案
4. Codex 按确认方案实现
5. 更新 docs 与 notes
6. 进行构建验证和人工验证

## 3. 模块实现方案必须包含的内容

- 模块目标
- 用户操作流程
- 文件结构
- 类型设计
- 页面状态设计
- service 方法设计
- View 组件职责
- 子组件职责
- props / emit 设计
- computed 使用点
- watch 使用点
- 关键函数职责
- 数据流 / 调用链
- mock 数据边界
- 后续接后端时的替换点
- notes 是否需要新增或更新

## 4. 状态设计规则

- 页面局部状态优先放在 View 组件中。
- 跨页面共享状态再考虑 Pinia。
- 当前 mock 数据优先通过 service 获取。
- 不要因为普通 CRUD 就引入全局 store。
- 表单组件应维护本地表单状态，不直接修改 props。

## 5. computed / watch 使用规则

- computed 用于派生数据，例如筛选后的列表、映射后的展示数据。
- watch 用于响应外部变化或执行同步/副作用，例如 props 变化后同步到弹窗本地表单。
- 不要用 watch 手动维护本该由 computed 推导的数据。
- 不要用 computed 承载需要被用户编辑的本地表单状态。

## 6. props / emit 使用规则

- 父组件负责页面状态、数据加载、调用 service。
- 子组件负责局部 UI 和表单交互。
- 父组件通过 props 给子组件传入数据。
- 子组件通过 emit 通知父组件事件。
- 子组件不直接修改 props。

## 7. service 方法设计规则

- service 负责 mock 数据和后续接口封装。
- View 不直接写大段 mock 数据。
- CRUD 逻辑优先放 service。
- 后续接后端时优先替换 service 内部实现。
- service 返回数据时应避免直接暴露内部可变数组。

## 8. 数据流说明模板

```text
用户操作
→ View 更新页面状态
→ 调用 service 方法
→ service 返回数据
→ View 更新响应式状态
→ computed 自动重新计算
→ 页面重新渲染
```

复杂模块需要写出自己的数据流，明确用户操作、状态更新、service 调用和页面渲染之间的关系。

## 9. 新模块设计草案模板

```md
# 模块实现方案：<模块名>

## 1. 模块目标

## 2. 用户操作流程

## 3. 文件结构

## 4. 类型设计

## 5. 页面状态设计

## 6. service 方法设计

## 7. 组件拆分

## 8. props / emit 设计

## 9. computed / watch 使用点

## 10. 关键函数与调用链

## 11. mock 边界与后端替换点

## 12. notes 更新计划
```

## 10. 适用范围

- Dashboard 这种简单展示页可以轻量设计。
- 提示词管理、模型配置、Prompt 调试台、测试记录、RAG、Agent Workflow 这类模块必须先写实现方案。
- 如果只是修样式或文案，不需要完整设计草案。
