# 模块设计流程规范

## 1. 文档目的

本文档用于约束新业务模块的设计流程。

- 任何复杂模块在写代码前，应该先明确数据流、状态设计、函数职责和组件通信方式。
- Codex 不应直接凭功能描述自由发挥实现模块。
- 模块实现应先形成可审核的方案，再进入代码阶段。

## 2. 标准开发流程

1. 明确模块目标。
2. 输出模块实现方案，不写代码。
3. 人类审核并确认方案。
4. Codex 按确认方案实现。
5. 更新 docs 与 notes。
6. 进行构建验证和人工验证。

## 3. 模块实现方案必须包含的内容

- 模块目标
- 本阶段范围
- 本阶段不做的内容
- 用户操作流程
- 页面信息结构
- 文件结构
- 类型设计
- 页面状态设计
- service 方法设计
- View 组件职责
- 子组件职责
- props / emit 设计
- computed 使用点
- watch 使用点
- 关键函数职责
- 数据流 / 调用链
- loading / empty / error 等页面状态
- mock 数据边界
- 后续接后端时的替换点
- 验收标准
- docs 是否需要更新
- notes 是否需要新增或更新
- 风险点与避免事项

## 4. 非目标范围与禁止事项

每个模块实现方案都必须明确本阶段不做什么。

- 不接真实后端接口，除非任务明确要求。
- 不真实调用大模型 API，除非任务明确要求。
- 不保存、展示或加密真实 API Key。
- 不引入新依赖。
- 不修改无关模块。
- 不修改路由、菜单、全局样式、构建配置，除非任务明确要求。
- 不为了“优化”重写已有模块逻辑。
- 不把计划中功能写成已完成能力。
- 如果任务需要突破以上限制，必须先说明原因和影响范围，等待确认后再执行。

## 5. 验收标准

每个模块方案必须写明验收方式。

- 页面访问路径。
- 核心交互验证。
- 是否需要执行 `cd frontend && npm run build`。
- mock 数据是否正常显示。
- 新增 / 编辑 / 删除 / 筛选 / 重置等操作是否按预期工作。
- docs 是否需要更新。
- notes 是否需要更新。
- 如果某项暂不验证，需要说明原因。

## 6. 状态设计规则

- 页面局部状态优先放在 View 组件中。
- 跨页面共享状态再考虑 Pinia。
- 当前 mock 数据优先通过 service 获取。
- 不要因为普通 CRUD 就引入全局 store。
- 表单组件应维护本地表单状态，不直接修改 props。

## 7. 页面状态补充规则

复杂模块设计时，需要判断是否需要以下状态：

- `loading`：数据加载或操作执行中。
- `empty`：列表为空、搜索无结果或暂无数据。
- `error`：接口失败或操作失败。
- `disabled`：按钮是否可点击。
- `dirty`：表单是否有未保存变更。

补充说明：

- 当前 mock 阶段可以轻量实现。
- 如果暂不实现，需要在模块方案中说明原因。
- 后续接后端时，`loading` / `error` 状态应优先补充。

## 8. computed / watch 使用规则

- computed 用于派生数据，例如筛选后的列表、映射后的展示数据。
- watch 用于响应外部变化或执行同步 / 副作用，例如 props 变化后同步到弹窗本地表单。
- 不要用 watch 手动维护本该由 computed 推导的数据。
- 不要用 computed 承载需要被用户编辑的本地表单状态。

## 9. props / emit 使用规则

- 父组件负责页面状态、数据加载、调用 service。
- 子组件负责局部 UI 和表单交互。
- 父组件通过 props 给子组件传入数据。
- 子组件通过 emit 通知父组件事件。
- 子组件不直接修改 props。

## 10. service 方法设计规则

- service 负责 mock 数据和后续接口封装。
- View 不直接写大段 mock 数据。
- CRUD 逻辑优先放 service。
- 后续接后端时优先替换 service 内部实现。
- service 返回数据时应避免直接暴露内部可变数组。

## 11. 命名规范

- 页面组件命名：`<Module>View.vue`。
- 弹窗组件命名：`<Module>Dialog.vue`。
- 业务局部组件放在：`src/views/<module>/components/`。
- service 文件命名使用小写业务名，例如 `prompt.ts`、`model.ts`。
- types 文件命名使用小写业务名，例如 `prompt.ts`、`model.ts`。
- 类型命名使用 PascalCase，例如 `PromptItem`、`ModelConfigItem`。
- 事件处理函数使用 `handleXxx`，例如 `handleCreate`、`handleSubmit`。
- 数据加载函数使用 `loadXxx`，例如 `loadPromptList`。
- 计算属性命名应表达结果含义，例如 `filteredPrompts`、`categoryLabelMap`。

## 12. 方案执行规则

模块方案确认后，代码实现必须遵循已确认方案。

- 模块方案确认后，Codex 不得自行改变关键设计。
- 如果实现过程中发现方案需要调整，必须先说明调整原因。
- 必须说明影响范围。
- 必须等待用户确认后再修改关键设计。

不得自行改变以下内容：

- 文件结构
- 状态位置
- service 方法命名
- props / emit 结构
- computed / watch 使用点
- 是否引入 Pinia
- 是否新增依赖
- 是否接入后端
- 是否调用真实模型 API

## 13. 版本化设计文档规则

模块设计文档应按阶段保存，避免把 mock、后端接入、RAG、Agent 等不同阶段混在同一份文档中。

建议结构：

```text
docs/modules/<module-name>/
├─ v1-mock-design.md
├─ v2-backend-integration.md
└─ v3-enhancement-plan.md
```

如果只是当前阶段的小范围调整，可以更新当前文档，例如：

- 增加字段。
- 修改筛选条件。
- 补 loading 状态。
- 调整局部交互。
- 修正 props / emit 说明。
- 修正文档描述错误。

如果模块的数据来源、调用链、职责边界发生变化，应新增新版本文档，例如：

- mock 数据改为后端接口。
- 前端筛选改为服务端分页筛选。
- 前端内存 CRUD 改为后端持久化 CRUD。
- mock LLM 输出改为真实模型调用。
- 普通 Prompt 测试改为 RAG 增强。
- 单步调用改为 Agent Workflow。

每份模块设计文档需要标明：

- 阶段：mock / backend / rag / agent / enhancement。
- 状态：草案 / 已确认 / 已实现 / 已废弃。
- 适用范围。
- 本阶段要做。
- 本阶段不做。
- 后续阶段入口。

如果新增新版本文档，需要在旧版本文档或 docs/modules/README.md 中标注后续阶段入口。

## 14. 模块文档目录建议

后续复杂模块的设计文档建议放在：

```text
docs/modules/
├─ README.md
├─ prompt/
│  ├─ v1-mock-design.md
│  └─ v2-backend-integration.md
├─ model-config/
│  ├─ v1-mock-design.md
│  └─ v2-backend-integration.md
└─ chat-test/
   └─ v1-mock-design.md
```

- `docs/modules/README.md` 用于做模块设计文档索引。
- 每个模块按阶段保存设计文档。
- 不建议把所有阶段追加到同一份巨大文档中。
- 历史设计文档不应随意删除，可以标注为“已废弃”或“被 v2 替代”。

## 15. 数据流说明模板

```text
用户操作
→ View 更新页面状态
→ 调用 service 方法
→ service 返回数据
→ View 更新响应式状态
→ computed 自动重新计算
→ 页面重新渲染
```

复杂模块需要写出自己的数据流，明确用户操作、状态更新、service 调用和页面渲染之间的关系。

## 17. 适用范围

- Dashboard 这种简单展示页可以轻量设计。
- 提示词管理、模型配置、Prompt 调试台、测试记录、RAG、Agent Workflow 这类模块必须先写实现方案。
- 如果只是修样式或文案，不需要完整设计草案。
