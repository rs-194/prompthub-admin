# docs 文档导航

本文件用于说明 `docs` 目录下各文档的用途、推荐阅读顺序和更新规则，帮助开发者在维护项目时快速找到对应文档。

根目录 `README.md` 面向项目展示，主要说明项目定位、技术栈、当前状态、启动方式和对外阅读信息；`docs/README.md` 面向文档导航，主要说明 `docs` 目录中文档之间的职责、阅读路径和维护规则。两者职责不同，不应互相替代。

## 1. 文档目录用途说明

`docs` 目录用于沉淀项目内部说明文档，包括项目概览、前端架构、路线图和开发记录等内容。

该目录下的文档应保持真实、可追溯，不应把计划中、待接入或仅有页面骨架的内容写成已完成能力。

## 2. 文档列表

| 文档 | 用途 | 需要更新的时机 |
| --- | --- | --- |
| `project-overview.md` | 说明项目背景、目标、核心模块、使用场景、MVP 范围和非目标范围。 | 项目定位、目标范围、核心模块、MVP 边界或非目标范围发生变化时更新。 |
| `frontend-architecture.md` | 说明前端目录结构、各目录职责、页面开发流程和后续模块放置规则以及技术债。 | 新增、删除、重命名目录或文件；调整页面、路由、菜单、布局、状态管理、类型、接口层或全局样式职责时更新。 |
| `roadmap.md` | 记录项目版本路线、计划内容和阶段状态。 | 新增业务规划、调整功能优先级、某个阶段状态从计划中变为已完成或待确认时更新。 |
| `development-log.md` | 持续记录开发过程中的关键变更，便于回溯每次改动的内容、影响范围和验证方式。 | 每次代码改动后追加记录；文档结构或项目维护规则发生重要变化时，可按需要记录。 |

## 3. 推荐阅读顺序

1. 先阅读根目录 `README.md`，了解项目展示信息、当前状态和启动方式。
2. 阅读 `project-overview.md`，确认项目背景、目标范围和当前阶段边界。
3. 阅读 `frontend-architecture.md`，了解前端目录职责和开发边界。
4. 阅读 `roadmap.md`，确认计划功能和阶段状态。
5. 阅读 `development-log.md`，回溯最近开发变更和验证记录。
6. 当前文件 `docs/README.md` 可作为文档导航入口，在不确定该读哪个文档时优先查看。

## 4. 文档更新规则

- 文档内容必须与项目真实状态一致，不编造不存在的接口、页面或能力。
- 计划中功能应明确保持为计划中，不写成已完成。
- 如果功能使用 mock 数据，需要明确标注当前为 mock 数据。
- 如果后端尚未接入，需要明确标注后端待接入。
- 如果只是页面骨架，需要明确标注页面骨架已完成，业务逻辑待实现。
- 修改目录结构、页面、路由、菜单、状态管理、类型、接口层、全局样式或构建方式时，应同步检查对应文档是否需要更新。
- 每次代码改动后，应至少检查 `development-log.md` 是否需要追加开发记录。
- 文档只记录已经确认的项目事实、计划和维护规则，不扩写尚未确认的业务能力。

## 5. README 职责边界

- 根目录 `README.md`：面向项目展示和快速了解，适合用于项目首页、作品集介绍和启动说明。
- `docs/README.md`：面向文档导航和维护协作，适合用于说明 `docs` 目录内各文档的用途、阅读顺序和更新规则。

两者可以互相引用，但不应重复承担同一职责。
